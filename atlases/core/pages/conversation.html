What this page will be

A staged resolver that funnels a vague human request into a concrete action plan, without asking one big LLM to do everything in one shot. Per the design discussion:

1. Cleanup & decomposition → goal, targets, exclusions, vague terms

"Find Mendelian inversions on chr12 but not from old families."
→ Restated as: goal=identify inversions with Mendelian-compatible inheritance; target=chromosome 12; exclusion=old families.

2. Domain selection → which keyword banks to load

Picks high-level domains from a controlled menu (genetics, relatedness, inheritance, structural_variation, population_history, …). Loads only those domains' keyword banks for the next stage — prevents the LLM from searching the whole vocabulary.

3. Keyword mapping → controlled vocabulary terms

Maps human words onto registry concepts: "mendelian" → analysis_id=mendelian; "old families" → set_filter=old_family_population_history_v1; "chromosome 12" → interval_set_id pattern.

4. Contract resolution → action plan or missing/spawnable list

Resolves the mapped vocabulary against the actual registries on disk. Outputs: required sets, required results, what's missing, what can be spawned, the recommended ordered chain of action manifests. The page-2 dashboard then visualizes the readiness ladder.

Why staged (not one-shot)

Each stage has a single job. The LLM doesn't jump from vague words to files. Stage outputs are inspectable, editable, and stable across sessions — so you can replay or tweak any stage without redoing the rest. The contract resolver is deterministic; only stages 1–3 are LLM calls.

Today, do this instead

  1. Use page 2 (Action) to pick a target analysis + scope. The dashboard walks the chain backward via analysis_modes.tsv and tells you what's RESULT_READY (reuse), RUN_READY, MISSING, or BLOCKED.
  2. Use page 3 (Registries) to see existing chains and which orphan results are ready to feed downstream analyses.
  3. Use scripts/resolve.py from the CLI for the same logic without the browser.
